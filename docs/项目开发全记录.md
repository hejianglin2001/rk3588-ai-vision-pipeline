# 1. 项目架构

```
RK3588 端侧 (C++)                        笔记本边缘侧
─────────────────                        ─────────
MIPI 摄像头 (IMX415, 1080p)
    │ UYVY
    ▼
V4L2 采集 (Multiplanar)
    │
    ▼ cv::rotate → resize → cvtColor
```

# 2. 阶段一: 模型转换

## 2.1 ONNX 导出

**目标:** yolo26n.pt → yolo26n.onnx

**问题:** `end2end=True` vs `end2end=False`

RK3588 NPU 不支持 NMS 算子。`end2end=True` 会把 NMS 也导出到 ONNX 里，RKNN 转换时 NMS 层会报错。

**解决:** 设置 `end2end=False`。ONNX 输出 `[1, 84, 8400]`——前 4 通道是已解码的 bbox，后 80 通道是 Sigmoid class scores。NMS 全部手写 C++。

```python
yolo.export(format="onnx", imgsz=640, opset=12, simplify=True)
# opset=12: RKNN-Toolkit2 最兼容的版本
# simplify=True: 用 onnxsim 合并冗余算子
```

## 2.2 FP16 vs INT8 选择

**初始方案:** FP16 (do_quantization=False)。精度 cos=0.998，class score 正常，推理 32ms (eval)。

**INT8 尝试:** 首次 INT8 量化后 class score 全零。详见 [阶段十](#11-阶段十-int8-量化攻坚)。

**结论:** 先用 FP16 跑通全链路，INT8 后期攻坚。

## 2.3 性能评估

```python
rknn.init_runtime(target="rk3588", perf_debug=True, eval_mem=True)
rknn.eval_perf(is_print=True, fix_freq=True)
```

结果: 31 FPS, NPU 全占用, CPU 几乎零。内存读写 58MB/帧。

# 3. 阶段二: C++ 推理管线

## 3.1 RKNN C API 学习

**问题:** 首次使用 `rknn_init` 传参错误，把文件路径字符串当模型二进制传。

**根因:** 文档不清晰。当 `size=0` 时 `rknn_init` 会把 model 指针当文件路径处理。

**解决:** `rknn_init(&ctx, model_path, 0, 0, NULL)` — size=0 触发文件路径模式。

## 3.2 预处理与输入格式

**问题:** ONNX 模型期望什么格式？

**解决:** `rknn_query(RKNN_QUERY_INPUT_ATTR)` 确认输入格式为 NHWC, UINT8, [1, 640, 640, 3]。
`pass_through=0` 让驱动自动做 `(pixel - mean) / std` 归一化（mean=[0,0,0], std=[255,255,255]）。

```cpp
input[0].type = RKNN_TENSOR_UINT8;
input[0].fmt  = RKNN_TENSOR_NHWC;
input[0].size = 3 * 640 * 640;
input[0].pass_through = 0;
```

## 3.3 输出反量化

**FP16 模型:** `want_float=1` 自动反量化为 float32。

**INT8 模型:** 见 [阶段十](#11-阶段十-int8-量化攻坚)。

## 3.4 NMS 后处理坐标转换

**问题:** bbox 输出是网格坐标还是像素坐标？

**根因:** YOLO `end2end=False` 导出时 `dist2bbox` 已解码——坐标在像素空间（网格坐标 × stride），不需要再乘 640。

**解决:** 直接使用原始值作为像素坐标：
```cpp
float x1 = cx - w / 2.0f;   // 不是 (cx - w/2) * 640
float y1 = cy - h / 2.0f;
```

**教训:** 在 `inspect_onnx.py` 打印前几个 anchor 的原始值，根据数值范围判断坐标系。

## 3.5 交叉编译环境

**编译器:** `/opt/gcc-arm-10.3-2021.07-x86_64-aarch64-none-linux-gnu/bin/aarch64-none-linux-gnu-g++`

**OpenCV:** 先用 SDK 自带精简版 (缺 highgui/videoio)，后换官方教材 4.7 完整版。

**RPATH 自包含部署:**
```cmake
set(CMAKE_INSTALL_RPATH "$ORIGIN/lib")
install(PROGRAMS ${OpenCV_LIBS} DESTINATION lib)
```

# 4. 阶段三: 摄像头采集

## 4.1 MIPI 摄像头识别

**问题:** 插上摄像头后 `/dev/video*` 出现但 OpenCV `VideoCapture(0)` 打不开。

**根因:** 内核 `imx415 4-001a: Unexpected sensor id` — DTS 配置了 IMX415 但可能插的 OV5695，或者排线接反。

**解决:**
1. 确认是 IMX415 (`dmesg | grep imx415`)
2. 翻面重插排线
3. 确认 `/dev/video11` 是 ISP 主路径输出

## 4.2 手写 V4L2 Multiplanar 采集

**问题:** 精简版 OpenCV 缺 videoio 模块，无法用 `cv::VideoCapture`。

**解决:** 手写 V4L2 驱动 (`camera_capture.h`)，使用 Multiplanar API：

```
ioctl(VIDIOC_S_FMT) → VIDIOC_REQBUFS → mmap → VIDIOC_QBUF → VIDIOC_STREAMON
→ VIDIOC_DQBUF → cvtColor(UYVY→BGR) → VIDIOC_QBUF
```

**关键细节:**
- Multiplanar: `V4L2_BUF_TYPE_VIDEO_CAPTURE_MPLANE`，不是单平面
- IMX415 ISP 输出格式: UYVY 4:2:2
- 分辨率: 最高 2592×1944, 实际使用 1920×1080

## 4.3 图像方向纠正

**问题:** 摄像头画面旋转 90°。

**解决:** `cv::rotate(frame, frame, cv::ROTATE_90_COUNTERCLOCKWISE)`

## 4.4 绿线花屏

**问题:** 断电重启后画面全是绿色条纹。

**根因:** MIPI 排线不支持热插拔——带电插拔导致 ISP 状态异常。ISP 默认输出格式在断电后可能漂移。

**解决:**
1. 完全断电重启（不是 reboot）
2. 强制设置 UYVY 格式: `fmt.fmt.pix_mp.pixelformat = V4L2_PIX_FMT_UYVY`

**问题:** 插显示器没画面。

# 6. 阶段五: JSON 输出与安全状态机

## 6.1 JSON 结构设计

**决策:** 输出结构化 JSON 包含 objects、obstacle、status，为 LLM 和日志系统服务。

**关键字段:**
- `objects[].bbox_2d / center_2d` — 2D 检测位置
- `distance_m: null` — 预留深度估计
- `obstacle.has_obstacle` — 安全状态机输出
- `status: normal/emergency_stop/disconnected` — 系统状态

## 6.2 安全兜底状态机设计

**核心原则:** 端侧硬实时检查，LLM 指令必须经过视觉验证。

**三条规则:**
1. `!connected` → 拒绝 LLM 指令，切本地避障
2. `MOVE_FORWARD` + 物体 `bottom > 85%` + `area > 40%` → 紧急刹车
3. `GRAB` + 无物体 → 拒绝

**关键参数:** 30%→40% 阈值调节，加入 `bottom > 85%` 条件避免远处物体误触发。

**面试话术:** "'建议者-审核者'架构——LLM 做高层规划，C++ 状态机做最终安全审核。每帧独立运行，不依赖网络。"

# 7. 阶段六: MQTT 端云通信

## 7.1 mosquitto 交叉编译

**问题:** 板端 `.so` 是 glibc 2.34，交叉编译器是 2.31，直接拷不能用。

**解决:** 从源码交叉编译 mosquitto + OpenSSL。

**步骤:**
1. 交叉编译 OpenSSL 3.0.12 → 静态库 `.a`
2. 建软链接让交叉编译器找到 OpenSSL: `sudo ln -s .../libssl.a .../lib64/`
3. 编译 mosquitto 2.0.18: `WITH_TLS=yes WITH_BUNDLED_DEPS=yes`

**关键坑:**
- mosquitto 子目录 Makefile 用系统 `cc` 而非交叉编译器 → `export CC/CXX/AR`
- OpenSSL 路径在 `lib/` 不是 `lib64/` → `LDFLAGS=/tmp/openssl_install/lib/libssl.a`
- `-lssl -lcrypto` 是 mosquitto 硬编码追加的 → 把 `.a` 放进交叉编译器 lib64

## 7.2 Broker 配置

**问题:** mosquitto 2.0+ 默认只接受本机连接。

**解决:**
```bash
echo -e "listener 1883 0.0.0.0\nallow_anonymous true" > /tmp/mqtt.conf
mosquitto -c /tmp/mqtt.conf
```

## 7.3 网络不通

**解决:** VMware 网络适配器切桥接模式，笔记本和板端同网段。

## 7.4 MQTT 日志系统

# 10. 阶段十: 多线程流水线

## 10.1 设计

```
T1 (采集): 摄像头 → rotate → resize → cvtColor → queue1
T2 (推理): queue1 → NPU → NMS → queue2
主线程:    queue2 → 安全 → 显示 → 性能报告
```

**关键设计:**
- `SafeQueue<T>` — 线程安全队列 (mutex + condition_variable)
- 背压: `max_size=4`，T1 产太快会阻塞
- `img.clone()` 深拷贝 — T1 和 T2 共享的 cv::Mat 必须 clone
- `std::move` 所有权转移 — queue 之间传递不拷贝
- 停止信号: `id=-1` 帧接力唤醒所有线程退出

## 10.2 死锁保护

**问题:** T1 退出时 T2 还在 `q1.pop()` 阻塞。

**解决:** T1 退出前推一个 `id=-1` 空帧。T2 检测到 `id<0` 后同样推空帧给主线程，实现级联优雅退出。

# 11. 阶段十一: INT8 量化攻坚

## 11.1 问题：class score 全零

**现象:** INT8 模型推理 class score 全为 0.000000，但整体余弦相似度 0.998。

**根因:** YOLO 输出 `[84, 8400]` 中 bbox(0~640) 和 class scores(0~1) 共享一个 INT8 量化 scale。bbox 数值大，scale≈2.71。Sigmoid 输出 ≤1 的值被 scale 碾为 0:
```
int8 = round(0.5 / 2.71) + (-128) = -128
float = (-128 + 128) × 2.71 = 0.000000  ← 消失
```

## 11.2 尝试过的无效方案

| 方案 | 结果 |
|------|------|
| `quantized_algorithm='kl_divergence'/'mmse'` | scale 仍被 bbox 主导 |
| 混合量化 (Sigmoid → float16) | Sigmoid 输出被 Concat 拉回 INT8 |
| 混合量化 (output0 → float16) | `output0_int8` Reshape 仍转回 INT8 |
| `quantized_dtype='w8a16'` | RK3588 不支持 |
| `output_tensor_type='float32'` | rknn-toolkit2 2.3.2 API 无此参数 |
| `auto_hybrid=True` | 未能保护 class score 通道 |

## 11.3 最终方案：ONNX 图手术——拆分输出

**思路:** 在 ONNX 图中最后一个 Concat 层之前切一刀，让 bbox 和 class scores 成为两个独立输出。INT8 量化时各自有独立 scale。

**实施:**
```python
# ONNX 图修改
# 最后 Concat 的 input = [Mul_2_output_0, Sigmoid_output_0]
# 加两个 Identity 节点引出两个新输出, 删掉原 Concat → output0

output_bbox  [1, 4, 8400]   ← bbox channels, scale ~2.5
output_score [1, 80, 8400]  ← class scores, scale ~0.004
```

**C++ 适配:** 推理时接收两个输出，拼接成 [84, 8400]，后处理逻辑不变。

## 11.4 结果

```
FP16 NPU:    50ms   FPS: 18.6
INT8 NPU:    25ms   FPS: 29.5   ✅ NPU 翻倍, class score 正常
```

**精度验证:**
- output_bbox  cos: 0.99868
- output_score cos: 0.98241

# 12. 阶段十二: RGA 硬件预处理 (已攻克 ✅)

## 12.1 概念

RGA (Raster Graphic Acceleration) = RK3588 的 2D 硬件加速器。
`imcopy(rga_buffer_t src, rga_buffer_t dst)` 可以在一次硬件调用中完成:
- resize (缩放)
- 颜色空间转换 (cvtColor)
- 旋转/翻转

**目标:** 用 RGA 替代 `cv::resize + cv::cvtColor`，CPU → RGA 硬件。

## 12.2 尝试 1: 虚拟地址模式 → 失败

**问题:** `wrapbuffer_virtualaddr_t` 模式下所有 `imcopy` 返回 `IM_STATUS_INVALID_PARAM (-3)`。

**根因:** kernel dmesg 显示 `page[nnn] failed to get pte`——RGA DMA 需要内存页常驻，但 `get_user_pages` 对 1920×1080×3(≈6MB) 的大缓冲区失败。

**尝试过的无效修复:**
- 16 字节 stride 对齐
- `madvise(MADV_WILLNEED)` — 建议内核预加载页
- `mlock()` — 锁定物理内存
- 格式转换 BGR→BGR (只缩放不换格式)

## 12.3 尝试 2: DMA-BUF fd 模式 → 失败

**思路:** 摄像头 V4L2 buffer 自带 DMA-BUF fd (内核 DMA 空间)，通过 `VIDIOC_EXPBUF` 导出，RGA 用 `wrapbuffer_fd_t(fd)` 直接读——零拷贝，绕开用户态页锁定。

**实施:**
1. `CameraCapture` 添加 `dma_fd_` 字段和 `VIDIOC_EXPBUF` 导出
2. `rga_preprocess.h` 添加 `src_fd` 参数，fd≥0 时用 `wrapbuffer_fd_t`
3. 源走 DMA-BUF (摄像头)，目标走虚拟地址 (640×640 够小)

**结果:** 仍然返回 `IM_STATUS_INVALID_PARAM (-3)`。

**根因分析:** RGA 硬件或内核驱动对 DMA-BUF 格式 (UYVY) + 缩放操作的组合支持不完整。独立 RGA 测试程序也失败，确认是驱动层问题，非应用代码问题。

## 12.4 当前状态

**代码状态:** ✅ 完整就绪
- `rga_preprocess.h` — 支持虚拟地址和 DMA-BUF 双模式
- `camera_capture.h` — 已添加 DMA-BUF 导出 (`get_fd()`, `get_pixfmt()`)
- `CMakeLists.txt` — 已链接 `librga.so`
- `main.cpp` — 已回退 OpenCV CPU 路径, RGA 调用保留但注释

**待解决:** 升级内核/driver 或使用 MPP 替代 (MPP 内部可能调 RGA)。

**面试话术:**
> "RGA 硬件加速的接入过程中遇到内核驱动层的 DMA 映射限制。我分别尝试了虚拟地址模式和 DMA-BUF 零拷贝模式，通过 dmesg 和 RGA 调试日志定位到 `get_user_pages` 失败是根因。已记录完整排查过程，待内核升级后启用。当前 CPU 路径虽慢但功能完备，不影响整体管线。"

## 12.5 调试记录

```bash
# 独立 RGA 测试 (排除应用层干扰):
# 640x480 → 640x640: INVALID
# 1280x720 → 640x640: INVALID  
# 1920x1080 → 640x640: INVALID (dmesg: get_user_pages failed)
# 640x640 → 640x640: FAILED (ret=0, 不是 1=SUCCESS)

# 内核日志:
rga: page[633] failed to get pte
rga: Only get buffer 2592768 byte from vma, but current image required 3690496 byte
```

**排查结论:** RK3588 RGA3 虚拟地址模式要求所有内存页在调用时已被访问过 (触发缺页中断)，大缓冲区跨多个页时内核无法一次 pin 住。DMA-BUF 模式理论上绕开此限制，但当前内核版本 (6.1.118) 的 RGA 驱动对 UYVY 格式的 fd 源支持不完整。

---

## 12.6 重试 3: posix_memalign + mlock → 全部通过 (2026-07-05)

**思路:** 之前的测试全部用的是 `malloc` (16 字节对齐)，RGA 的 DMA 需要页对齐内存。换成 `posix_memalign` (4096 字节页对齐) + `mlock` 锁定物理内存后再传 `wrapbuffer_virtualaddr`。

**测试程序:** `tools/test_rga.cpp` — 逐级验证，从最小缓冲到最大缓冲：

```
【第1级】BGR 320x240 → RGB 640x640        ✅ OK
【第2级】BGR 1920x1080 → BGR 640x640      ✅ OK  ← 摄像头尺寸!
【第3级】BGR 1920x1080 → RGB 640x640      ✅ OK  ← 缩放+格式转换!
【第4级】UYVY 1920x1080 → RGB 640x640     ✅ OK  ← 真实摄像头格式!
```

**关键发现:** 之前失败不是内核驱动限制，是**内存分配方式不对**。`malloc` 分配的内存不保证页对齐，`get_user_pages` 失败是因为页边界跨了不连续物理页。`posix_memalign(4096)` 保证页对齐 + `mlock` 防止被 swap 出去，RGA 的 DMA 就能正确锁住所有页。

**集成到主线:**

1. 在 `camera_capture.h` 添加 `read_raw()` 方法 — 不经过 CPU 颜色转换，直接返回内核 DMA 缓冲区指针
2. 重写 `rga_preprocess.h` — 用 `posix_memalign` + `mlock` 模式，`imresize` 一步完成 UYVY→RGB + 1920×1080→640×640
3. T1 采集线程改为：
   ```
   cam.read_raw() → RGA(UYVY→RGB 640×640) → cam.qbuf() → rotate(640×640) → clone → queue
   ```

**效果:** 采集耗时从 18ms 降到 **4.2ms**（↓76%）。CPU 省掉 `cvtColor(UYVY→BGR, 6MB)` + `resize(6MB→1.2MB)` + `cvtColor(BGR→RGB, 1.2MB)`，全部由 RGA 硬件替代。

**面试话术:**
> "RGA 硬件加速的接入过程遭遇了内核 DMA 页锁定失败。我通过最小测试程序逐级排查，最终定位到根因是 `malloc` 分配的内存不满足页对齐要求。改用 `posix_memalign(4096)` + `mlock` 后，RGA 在 1920×1080 UYVY → 640×640 RGB 的缩放+色彩转换上稳定运行，采集耗时从 18ms 降至 4.2ms，降幅 76%。"

# 13. MPP 硬编码与 RTSP 推流

## 13.1 背景与目标

**目标:** 将 YOLO 推理结果 (带检测框的画面) 通过 RTSP 协议实时推流，使局域网内任何设备 (PC、手机、NVR 录像机) 都能通过标准 RTSP URL 观看。

**为什么需要 MPP？**

RK3588 有独立 VPU (Video Process Unit) 硬编码器，专做 H.264/H.265 编码。用 CPU 软件编码 (如 libx264) 会挤占 NPU 推理的资源，导致掉帧。MPP (Media Process Platform) 是瑞芯微官方的多媒体框架，直接调用 VPU 硬件。

**为什么要 RTSP 而不是原始 UDP/TCP？**

RTSP (Real Time Streaming Protocol) 是视频监控行业的标准协议：
- 客户端主动发起 DESCRIBE → SETUP → PLAY，服务器按需推流
- 支持多客户端同时观看
- VLC、ffplay、NVR、浏览器 (通过 WebRTC 转) 都能直接打开 `rtsp://ip:8554/live`

---

## 13.3 尝试过程与踩坑记录

### 尝试 1: GStreamer → UDP RTP → PC SDP 文件 (部分成功)

**思路:** GStreamer 直接用 `mpph264enc → rtph264pay → udpsink` 把 RTP 包打到 PC 的 UDP 端口。

```bash
gst-launch-1.0 filesrc location=/tmp/video_pipe ! \
    video/x-raw,format=NV12,width=640,height=640,framerate=30/1 ! \
    mpph264enc ! h264parse ! rtph264pay ! \
    udpsink host=192.168.0.104 port=5600
```

PC 端创建 SDP 文件描述流参数，VLC/ffplay 打开。

**结果:** ffplay 能播放但报 `non-existing PPS 0 referenced`。

**根因:** H.264 解码必须知道 SPS (序列参数集: 分辨率、profile) 和 PPS (图像参数集: 编码模式)。`rtph264pay` 默认只在流开头发一次 SPS/PPS，ffplay 半路接入时错过了，永远解不出第一帧。

**部分解决:** 加 `config-interval=1` 让 RTP payloader 每秒重发一次 SPS/PPS。ffplay 1 秒内拿到参数开始解码。

**为什么没继续用 UDP:** 这不是标准 RTSP。客户端需要手动 SDP 文件，VLC 装 live555 插件才能打开。不适合"给面试官一个 URL 就能看"的目标。而且 UDP 丢包不可恢复，局域网虽然概率低但无法处理网络抖动。

### 尝试 2: GStreamer → RTSP Server (失败)


**结果:** `test-launch` 命令不存在。板端有 RTSP 库但没有工具包。尝试 `apt install libgstrtspserver-1.0-dev` 自己编译，但 glib 依赖版本冲突 (`2.72.4-0ubuntu2.9` vs `2.72.4-0ubuntu2.6`)，apt 报 broken packages 无法安装。

**结论:** 板端包管理不够干净，源码编译 RTSP server 不划算。

### 尝试 3: mediamtx + UDP RTP 直读 (失败)

**思路:** mediamtx 是 GitHub 25k+ star 的企业级 RTSP 服务端 (Go 单文件)，配置它直接读取 GStreamer 输出的 UDP RTP 包，转成 RTSP 给客户端。

```yaml
# mediamtx.yml
paths:
  live:
    source: rtp://127.0.0.1:5600
```

**结果:** `ERR: invalid source: 'rtp://127.0.0.1:5600'`。mediamtx v1.11.3 版本配置语法变了，旧版 `source: rtp://` 已废弃。

尝试 `source: rtpVideo` + `rtpVideoAddress: :5600` → `ERR: unknown field "rtpVideoAddress"`。

反复试了 `rtpVideo`、`rtp`、`rtsp` 等字段名，都不认。

**结论:** mediamtx v1.11.3 的配置文档和实际行为有出入，不再死磕配置。

### 尝试 4: mediamtx + ffmpeg 推流 (成功 ✅)

**思路:** 不让 mediamtx 直接读 RTP。改用 `source: publisher` 模式——mediamtx 只管理客户端连接，ffmpeg 负责把 H.264 流推给它。

```
GStreamer (MPP编码) → H.264 TCP → ffmpeg (协议转换) → RTSP 推流 → mediamtx (服务端)
```

```yaml
# mediamtx.yml
paths:
  live:
    source: publisher  # 接受外部推流
```

```bash
# ffmpeg 读本地 H.264 TCP → 推到 mediamtx
ffmpeg -re -f h264 -probesize 32 -i tcp://127.0.0.1:5601 \
    -c copy -f rtsp rtsp://127.0.0.1:8554/live
```

**关键参数:**
- `-f h264`: 告诉 ffmpeg 输入是 H.264 Annex B 裸流 (否则不认识)
- `-probesize 32`: 不等待大文件分析，立即解析 (管道流没有文件头)
- `-c copy`: **不重新编码**。H.264 数据原样转发，只换容器格式 (H.264 raw → RTSP 封装)。这是关键——如果这里写 `-c:v libx264`，CPU 会重新编码一遍，那 MPP 硬件编码就白做了
- `-re`: 按帧率速度读取 (realtime)，不加会一口气读完管道

**还踩了一个坑:** ffmpeg 后台运行被 SIGTSTP 暂停 (`[Stopped]`)，因为后台进程读 TCP socket 时被终端信号干掉。修复：`nohup ... </dev/null &` 彻底脱离终端。

## 13.5 核心技术概念 (面向初学者)

### 什么是 FIFO (命名管道)？

FIFO 是 Linux 内核提供的一种特殊文件。它像一个水管——一头写、一头读，数据不落盘。

```bash
mkfifo /tmp/video_pipe   # 创建一个"水管"
echo "hello" > /tmp/video_pipe &   # 往水管写 (没人读会阻塞)
cat /tmp/video_pipe      # 从水管读
```

在我们的系统里：
- C++ 程序是"写端"：`fwrite(nv12_data, 1, size, stream_pipe)`
- GStreamer 是"读端"：`filesrc location=/tmp/video_pipe`
- 好处：C++ 和 GStreamer 是两个独立进程，不共享内存，FIFO 天然提供了背压 (写太快了就阻塞)

### 为什么 GStreamer 命令行而不是 C++ 代码？

当前阶段用 `gst-launch-1.0` 命令行。生产环境最终应该用 GStreamer C API 嵌入到 main.cpp 中：

```cpp
// 下一阶段: 进程内 pipeline, 去掉 FIFO
GstElement *pipeline = gst_parse_launch(
    "appsrc name=src ! videoparse ... ! mpph264enc ! ...", NULL);
// 每帧: gst_app_src_push_buffer(appsrc, nv12_buffer);
```

但命令行版本已经足够验证整个链路，而且调试方便 (改参数不用重编译)。

### 什么是 H.264 Annex B？

H.264 码流有两种封装格式：
- **Annex B**: 每个 NAL 单元以 `00 00 00 01` 起始码分隔，裸流/文件常用
- **AVCC**: 每个 NAL 单元前有长度字段，MP4/FLV 容器用

`mpph264enc` 输出 Annex B 格式。`tcpserversink` 原样转发。ffmpeg `-f h264` 期望 Annex B 输入。整条链路格式一致，所以 `-c copy` 零开销。

### 什么是 SPS/PPS？

H.264 码流里有特殊参数帧：

| 缩写 | 作用 | 例子 |
|------|------|------|
| **SPS** (Sequence Parameter Set) | 序列级参数 | 分辨率 640×640、High Profile |
| **PPS** (Picture Parameter Set) | 图像级参数 | 熵编码 CABAC、去块滤波参数 |
| **IDR** (Instantaneous Decoder Refresh) | 关键帧 | 完整的独立图像，不解码前面的帧也能显示 |

解码器必须先收到 SPS+PPS 才能开始解码。这就是为什么之前 UDP RTP 直推时 ffplay 报 `non-existing PPS`——它错过了开头的 SPS/PPS。

**RTSP 的优势**: mediamtx 的 RTSP 协议在客户端 DESCRIBE 时会重新发送 SPS/PPS，所以任何时候接入都能立刻解码。

## 15.3 关键发现: DDR 内存总线是隐藏瓶颈

RK3588 的 DDR 子系统（4-ch LPDDR5, 峰值 34GB/s）带宽充裕，但所有硬件模块（RGA、NPU、VPU、CPU）共享同一个 DDR 控制器。控制器按优先级和 bank 冲突调度请求，多路同时访问时互相等待。每开一个模块就是往控制器上加一个竞争者。

### 发现 1: 单线程 = 各模块独立最快

```
单线程串行:  RGA(3.6ms) → NPU(23ms) → NMS(4ms) → NV12(6ms) → 39ms/帧 ≈ 25.6 FPS
              ↑独占DDR     ↑独占DDR    ↑独占DDR   ↑独占DDR
```

每个模块独占总线，跑出真实速度。

### 发现 2: 多线程 = 模块变慢但吞吐提高

```
多线程并行:  RGA ────┐
            NPU+NMS ─┼── 三路抢DDR → 每一路都变慢, 但时间重叠
            NV12 ────┘   → 33ms/帧, 3路重叠 → 30 FPS
```

capture 从 3.6ms 涨到 33ms（等 9 倍），但 FPS 从 25 上到 30——"牺牲延迟换吞吐"。这就是多线程本质：**不是让每个模块变快，是让它们同时干活，总时间 = max(各阶段) 而不是 sum**。

### 发现 3: RGA 在多线程下变成负优化

多线程下关了 RGA，capture 从 33ms 降到 7ms——RGA 的 6MB DMA 传输（1920×1080 UYVY）在 DDR 上是"大户"，优先级高、占用时间长。把它换成 CPU resize，DDR 少一个大竞争者，系统整体更和谐。

### 发现 4: 零拷贝改变 DDR 抢占时序

加了零拷贝（配置 I），NMS 从 15.5ms 降回 3.7ms，capture 涨到 33ms。`memcpy` 到 NPU DMA 内存在忙碌检测上赢了 NMS 的 `postprocess_split`。零拷贝值不值得开，取决于你的瓶颈在哪。

### 发现 5: 多核 NPU 并行对单流无用

尝试用 `RKNN_NPU_CORE_0_1_2` 三核并行、6 线程轮转——FPS 从 30 跳到 60 又跌回 18，剧烈波动。三核自动分配对单路流是过度抢占，DDR 带宽直接过载。

## 15.4 结论: 企业生产的模块选择逻辑

**不是全开，是按瓶颈针对性开启：**

| 模块 | 开/关 | 条件 |
|------|------|------|
| **RGA** | 开启 | 单线程必开（capture ↓50%） |
| **多线程** | 关闭 | NPU 25ms 是瓶颈，开了改 DDR 抢占但总吞吐提升 5FPS |
| **RTSP** | 根据需要 | 网络输出需要才开，不输出就关 |
| **零拷贝** | 可开 | 单线程下无副作用 |
| **多核 NPU** | 关闭 | 单流没用，多流才考虑 |

**最优交付配置: RGA=1, MT=0, RTSP=1, ZCOPY=1 → 26.3 FPS, capt=3.6ms, NPU~23ms, NMS~4ms**

NPU 25ms 是物理天花板（INT8 量化后），FPS 上限 ~40。再往上的优化方向是换 ReLU 模型或用更轻的 backbone。

# 17. 面试核心话术

## 17.1 INT8 量化

> "YOLO 检测头的 bbox 和 class score 共享输出张量，INT8 量化时共享 scale 导致 Sigmoid 小数值被碾为零。我通过 ONNX 图手术将输出拆分为两个独立张量，各自享有独立的量化 scale。NPU 推理耗时减半且精度无损。这是量化工程中'动态范围不匹配'问题的标准解法。"

## 17.2 多线程管线

> "端侧推理管线采用三线程流水线架构——采集、推理、后处理并行。利用 std::queue + condition_variable 实现背压控制的线程安全队列，img.clone 深拷贝保证数据隔离，std::move 所有权转移减少不必要拷贝。吞吐量从 15FPS 提升到 30FPS。"

## 17.3 安全兜底

> "'建议者-审核者'安全架构——LLM 做高层任务规划，C++ 状态机做最终硬实时审核。每帧独立运行的视觉检测不依赖网络状态，即使 LLM 产生幻觉指令，端侧也能强制刹车。断网时自动降级为纯视觉避障模式。"

## 17.4 交叉编译与部署

> "从源码交叉编译了 mosquitto 和 OpenSSL，解决 glibc 版本不匹配问题。采用 RPATH $ORIGIN/lib 实现自包含部署，无需修改板端系统库。构建脚本一键编译并打包所有依赖，开发迭代周期 <30 秒。"

## 17.5 RGA 硬件加速

> "RGA 硬件加速的接入过程遭遇了内核 DMA 页锁定失败。我通过最小测试程序逐级排查——从 320×240 小缓冲到 1920×1080 摄像头尺寸——最终定位到根因是 `malloc` 分配不满足页对齐要求。改用 `posix_memalign(4096)` + `mlock` 后，RGA 在 UYVY 1920×1080→RGB 640×640 的缩放+色彩转换上稳定运行。采集耗时从 18ms 降至 4.2ms，降幅 76%。这个排查过程体现了嵌入式调试的系统方法——从硬件特性反推软件要求。"

## 17.6 MPP 硬编码与 RTSP 流媒体

> "RK3588 的 VPU 硬编码器通过 MPP 框架接入。我设计的生产链路是：C++ 进程写 NV12 到 FIFO → GStreamer mpph264enc 硬件编码 → ffmpeg 协议转换 → mediamtx 企业级 RTSP 服务端。尝试过 4 种方案——UDP 裸 RTP、GStreamer RTSP Server 自编译、mediamtx RTP 直读、mediamtx publisher 模式——最终定型为 MPP+ffmpeg+mediamtx 的铁三角架构。全程 `-c copy` 零重编码，SPS/PPS 通过 RTSP DESCRIBE 自动协商，客户端 VLC 直接 `rtsp://ip/live` 打开。这就是 AI IPC/智能摄像头的标准生产推流方案。"

## 17.7 性能对比方法论

> "我为项目设计了完整的 A/B 测试框架——3 个编译期开关（USE_RGA/USE_MULTITHREAD/USE_RTSP）组合出 8 种配置，MQTT 自动上报性能指标，logger 按配置自动命名日志文件。我实际跑了全部 8 种组合，发现单线程+RGA 比多线程+CPU 的采集快 50%、NMS 更稳定。这个结论反直觉——多线程没有带来吞吐提升，因为 NPU 25ms 是绝对瓶颈。工程决策不能靠直觉，要用数据说话。"
